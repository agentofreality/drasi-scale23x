The Problem

Detecting and Reacting to Change is Harder Than It Should Be

This is the premise of the entire talk. Pause here — everyone in this room has dealt with this problem. Let the statement resonate before continuing.

A Familiar Scenario

  • You have data spread across multiple systems
  • Data changes (or doesn't change when expected)
  • Your system needs to detect this and react

Quickly. Precisely. Reliably.

Data lives in databases, APIs, Kubernetes, SaaS platforms — everywhere. Note: "or doesn't change when it should" — absence of change is just as important. The requirements sound simple, but the implementation rarely is.

Example: Proactive Customer Care

A shipping carrier marks a delivery as "delayed" — your e-commerce platform needs to identify affected customers, check their loyalty tier, and offer compensation before they complain

Key point: the carrier system knows nothing about your customers or their loyalty tier. The reaction requires joining data across system boundaries — this is where it gets hard. This example highlights cross-source data correlation and the need for speed.

Example: Field Service Dispatch

Maintain a live set of technicians who are near an open high-priority ticket, certified for the equipment, and not already dispatched — across GPS, ticketing, HR, and dispatch systems

This is a dynamic result set whose membership changes constantly across four systems. The challenge isn't finding qualified technicians once — it's maintaining a live view as GPS positions update, tickets change status, and dispatches happen.

Example: AI Support Agent

A customer calls about a missing order. Your AI support agent searches its knowledge base — but the embeddings were last refreshed overnight, so it doesn't know the order was rerouted two hours ago. The agent confidently gives the wrong answer.

Ooops. What if the vector DB updated the moment the order status changed — not on the next batch run?

This example highlights a real problem with RAG-based AI agents: vector databases are typically updated in periodic batches — hourly, nightly. Between refreshes, the agent's knowledge is stale. Drasi enables incremental, immediate updates to the vector DB the moment source data changes. The agent always has fresh context. This reframes change-driven architecture as an AI infrastructure concern, not just a backend automation pattern.

Building Change-Driven Systems

How Teams Do It Today

Now let's look at the common approaches teams use to build change-driven systems. We'll examine three: polling, analytics platforms, and CDC + stream processing. Each has trade-offs — and all require significant custom engineering.

Three Common Approaches

  1. Polling
  2. Analytics Platforms
  3. Event Stream Processing
Quick overview — three approaches teams commonly use. We'll look at each briefly before examining the shared problems they all have.

Approach 1: Polling

Your system queries the data store periodically, compares the results to what it saw previously, and acts on the differences.

Polling a database is like opening the fridge every five seconds to see if new food appeared

Polling is everyone's first solution — intuitive, no new infrastructure needed. Query, compare to last time, act on differences. The fridge analogy gets a laugh and makes the inefficiency tangible. Pause on it before moving to the next approach.

Approach 2: Analytics Platforms

Bring data together in a central analytics platform where rich queries and dashboards can surface meaningful conditions across all your data.

Analytics platforms solve the multi-source query problem elegantly — rich queries across all your data in one place. But the model is fundamentally retrospective: you're analyzing a copy of data that's already minutes or hours stale.

Approach 3: Event Stream Processing

CDC reads row-level changes from the data store and streams them to an event processing engine for filtering, joining, transformation, and change detection.

This is genuinely powerful, battle-tested technology. We're not dismissing it. But these are event-driven tools — excellent at moving events at scale. Building change-driven logic on top still requires substantial engineering.

The Problems

Dev and Ops complexity, focus on plumbing not business solution, often one-off brittle solutions...

Polling

  • Poll interval defines detection lag
  • Many polls return nothing
  • You own state & restarts
  • Every use case = new loop
  • Doesn't scale

Analytics

  • Cost prohibitive
  • Querying stale data
  • ETL lag: minutes to hours
  • Pipelines & schema sync
  • Insight, not reaction

Event Stream

  • Heavy infrastructure stack
  • Consumer owns all logic
  • Cross-source joins are hard
  • Schema evolution ripples
  • Weeks to production

Introducing

Now let's meet Drasi. We've established the problem and the limitations of current approaches. Drasi is an open-source platform designed specifically for building change-driven architectures.

What is Drasi?

Data change processing platform for building change-driven solutions.

Open Source · Apache 2.0 · CNCF Sandbox

Drasi is a Data Change Processing platform — not event streaming, not stream processing. It uses declarative continuous queries to isolate meaningful data changes and trigger reactions. CNCF Sandbox since January 2025, Apache 2.0 licensed.

Core Concepts

Three simple concepts are the foundation of everything Drasi does

Pause here — let the simplicity land. Source. Continuous Query. Reaction. Sources connect to your existing data. Queries define what changes matter. Reactions act when results change. Three concepts, that's the entire platform.

Sources

  • Manage connection (and reconnection) to data source
  • Receive / retrieve change events or mimic change feed
  • Transform change events to property graph node/relation updates
  • Emit Source Change Events to subscribed Continuous Queries
Source Change Events are the internal data format that Sources use to communicate changes to Continuous Queries. Three variants: Insert (new element with full data), Update (changed element with full data), Delete (only metadata — identity and labels, no properties needed since the element is gone). Key point: the Source handles all the CDC complexity — reading WAL logs, tracking offsets, mapping rows to graph elements. By the time data reaches the Continuous Query, it's a clean, typed change event.

Continuous Query

  • Subscribe to one or more Sources and receive Source Change Events
  • Declarative openCypher or GQL queries describe the change you are interest in
  • Maintain perpetually accurate Result Set
  • Send Result Change Notifications containing diff to subscribed Reactions
This diagram shows what happens inside a Continuous Query. Source Change Events (Insert, Update, Delete) arrive from Sources. The query engine evaluates each change against the query, updating three internal indexes: Element Index — stores all Nodes and Relations the query subscribes to. Aggregation Index — maintains running aggregations (count, sum, avg, etc.) so they don't need to be recomputed from scratch. Result Index — the current result set that satisfies the query. After processing a change, the engine diffs the result set and emits Result Change Notifications: Added (new row matches), Updated (row still matches but values changed, with before/after), Deleted (row no longer matches). These typed notifications flow to Reactions — clean, unambiguous, no parsing required.

Example: Continuous Query

All stock prices


MATCH (sp:stock_prices)
RETURN sp.symbol AS symbol,
       sp.price AS price,
       sp.previous_close AS previousClose,
       ((sp.price - sp.previous_close) / sp.previous_close * 100) AS changePercent
symbol price previousClose changePercent

MSFT412.35415.80-0.83

Simple example — match all stock prices and compute the change percentage. This query runs perpetually. Every time a price updates, the result set changes and Drasi emits the diff. Point out: this looks like a normal query, but it's live — not a one-time SELECT.

Example: Continuous Query

All stocks with a price higher than yesterday's close


MATCH (s:stocks)-[:HAS_PRICE]->(sp:stock_prices)
WHERE sp.price > sp.previous_close
RETURN s.symbol AS symbol,
       s.name AS name,
       sp.price AS price,
       sp.previous_close AS previousClose,
       ((sp.price - sp.previous_close) / sp.previous_close * 100) AS changePercent
symbol name price previousClose changePercent
AAPLApple Inc.189.42187.10+1.24
NVDANVIDIA Corp.875.60860.20+1.79
AMZNAmazon.com182.75180.90+1.02
Now we add a relationship traversal and a WHERE filter. This query joins stocks with their prices and only returns gainers. As prices change, stocks enter and leave this result set automatically. Highlight the MATCH pattern — this is graph query syntax, not SQL JOINs.

Reactions receive typed change events: added (new result matches), updated (result changed but still matches, with before/after), deleted (no longer matches). Clean, unambiguous — no parsing event payloads to figure out what happened. Flexible wiring: one query can feed many reactions, one reaction can watch many queries. Absence of Change

  • Detect what didn't happen
  • trueFor — condition persists for duration
  • trueLater — condition is true at a specific time
  • trueUntil — condition remains true until a specific time

Temporal Functionality

  • Compare a changed element with its previous values
  • Process a range of previous values for the changed element
  • Get the Continuous Query result as it was at any point in time

Change-driven UI

  • Drive UI content from Continuous Queries
  • Drop in React and Vue components display Continuous Query Result
