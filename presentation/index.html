The Problem

Detecting and Reacting to Change is Harder Than It Should Be

This is the premise of the entire talk. Pause here — everyone in this room has dealt with this problem. Let the statement resonate before continuing.

A Familiar Scenario

  • You have data spread across multiple systems
  • Data changes (or doesn't change when it should)
  • Your system needs to detect this and react

Quickly. Precisely. Reliably.

Data lives in databases, APIs, Kubernetes, SaaS platforms — everywhere. Note: "or doesn't change when it should" — absence of change is just as important. The requirements sound simple, but the implementation rarely is.

Example: Proactive Customer Care

A shipping carrier marks a delivery as "delayed" — your e-commerce platform needs to identify affected customers, check their loyalty tier, and offer compensation before they complain

Key point: the carrier system knows nothing about your customers or their loyalty tier. The reaction requires joining data across system boundaries — this is where it gets hard.

Example: Supply Chain Deadlines

A supplier committed to delivering parts by a deadline, but the warehouse receiving system hasn't logged an arrival — production planning needs to adjust the schedule before the line stops

This is "absence of change" — detecting something that DIDN'T happen by a deadline. No single system owns this logic. Traditional event-driven systems struggle here because there's no event to consume when nothing happens.

Example: Field Service Dispatch

Maintain a live set of technicians who are near an open high-priority ticket, certified for the equipment, and not already dispatched — across GPS, ticketing, HR, and dispatch systems

This is a dynamic result set whose membership changes constantly across four systems. The challenge isn't finding qualified technicians once — it's maintaining a live view as GPS positions update, tickets change status, and dispatches happen.

Sound Familiar?

These are all examples of Change-Driven Systems:

Systems that take action in response to a meaningful change — or the absence of change — in relevant data.

A problem you've probably already solved...

... more than once!!

... using different techniques!!

This definition captures two critical concepts: "meaningful change" (not every row update) and "absence of change" (detecting what didn't happen). Both are first-class concerns. Build the fragments slowly — let the audience recognize themselves in this pattern.

Building Change-Driven Systems

How Teams Do It Today

Now let's look at the common approaches teams use to build change-driven systems. We'll examine three: polling, analytics platforms, and CDC + stream processing. Each has trade-offs — and all require significant custom engineering.

Approach 1: Polling

Your system queries the data store periodically, compares the results to what it saw previously, and acts on the differences.

Polling is everyone's first solution — intuitive, requires no new infrastructure. Query, compare to last time, act on differences. Simple mental model. But as you'll see, this simplicity hides real problems that emerge at scale.

Polling: The Problems

Deceptively easy, but problems often emerge:

  • Latency — Detection lag equals the poll interval; faster reaction requires more frequent polling
  • Wasted Resources — Low velocity data, high frequency polling, and large datasets make polling inefficient
  • State Management — Where to store previous results? How to manage schema changes? What about restarts and resilience?
  • Multiple Solutions — Each use case requires new and usually unique implementation
  • Doesn't Scale - Multiple solutions, integrating multiple data sources, and large datasets all increase complexity significantly
The polling interval is a false economy — too slow means missed changes, too fast means wasted resources. State management is often overlooked until restarts cause duplicate processing. And N consumers × M sources = N×M polling loops.

Approach 2: Analytics Platforms

Bring data together in a central analytics platform where rich queries and dashboards can surface meaningful conditions across all your data.

Analytics platforms solve the multi-source query problem elegantly — rich queries across all your data in one place. But the model is fundamentally retrospective: you're analyzing a copy of data that's already minutes or hours stale.

Analytics Platforms: The Problems

  • Data is Copied, Not Live — Your analytics platform holds a snapshot; the real systems have already moved on
  • ETL Latency — Data pipelines introduce minutes to hours of lag before changes are visible
  • Retrospective, Not Reactive — Designed for answering "what happened?" — not "act when this changes"
  • Operational Overhead — Maintaining ETL pipelines, schema mappings, and sync schedules is a job in itself
Analytics platforms excel at "what happened?" — historical analysis over large datasets. But change-driven systems need "act when this changes" — fundamentally different. You can't react to a change that hasn't been loaded through ETL yet.

Approach 3: CDC + Stream Processing

Change Data Capture reads row-level changes from the data store and streams it in real time to an event / stream processing engine for filtering, joining, and transformation. The dominant approach in serious production systems today.

A rich, mature, battle-tested ecosystem:

Debezium Apache Kafka Kafka Connect Apache Flink Apache Spark Streaming AWS Kinesis Azure Event Hubs Google Pub/Sub Confluent Platform Materialize ksqlDB RisingWave
This is genuinely powerful, battle-tested technology. We're not dismissing it. But these are event-driven tools — excellent at moving events at scale. Building change-driven logic on top still requires substantial engineering.

CDC + Stream Processing: In Practice

Count the boxes — each is a system to deploy, configure, monitor, and maintain. Each arrow is a contract: schemas, serialization formats, consumer offsets. And we haven't even discussed the consumer logic: filtering, joins, deduplication.

The Complexity Cost

Common approaches to building change-driven systems today require substantial engineering, configuration, and ongoing maintenance — all to express what should be simple logic about data changes and how to react.

The key insight: most of this complexity isn't your business logic — it's plumbing. Engineering time goes to infrastructure instead of expressing what changes matter. Cross-source joins are especially painful — windowing, state stores, exactly-once.

There has to be an easier way !

What we really need is something that is:

  • Expressive & Live — Define what matters, react as it happens
  • Reliable & Scalable — Handles state, restarts, and growth
  • Low Ops — Minimal infrastructure, works with existing systems

No vendor lock-in, community-driven.

These are the characteristics we need. Expressive queries over live data. Reliable handling of state, restarts, schema changes. Minimal ops burden. Open source with no vendor lock-in. This sets up what Drasi provides.

Introducing

Now let's meet Drasi. We've established the problem and the limitations of current approaches. Drasi is an open-source platform designed specifically for building change-driven solutions.

What is Drasi?

Open source platform for building change-driven solutions.

CNCF Sandbox · Apache 2.0

Drasi is a Data Change Processing platform — not event streaming, not stream processing. It uses declarative continuous queries to isolate meaningful data changes and trigger reactions. CNCF Sandbox since January 2025, Apache 2.0 licensed.

Core Concepts

Pause here — let the simplicity land. Source. Continuous Query. Reaction. Sources connect to your existing data. Queries define what changes matter. Reactions act when results change. Three concepts, that's the entire platform.

Sources

Connect to your existing data

PostgreSQL MySQL SQL Server Cosmos DB Kubernetes Event Hubs Dataverse

No new databases or queues required — connects to what you already run

Sources connect to your existing infrastructure — no rip-and-replace required. PostgreSQL, MySQL, SQL Server for relational. Cosmos DB Gremlin for graph. Kubernetes for cluster state. Event Hubs for streaming. Dataverse for Microsoft ecosystem.

Source Change Events

Sources emit Source Change Events to notify Continuous Queries of data changes

Source Change Events are the internal data format that Sources use to communicate changes to Continuous Queries. Three variants: Insert (new element with full data), Update (changed element with full data), Delete (only metadata — identity and labels, no properties needed since the element is gone). Key point: the Source handles all the CDC complexity — reading WAL logs, tracking offsets, mapping rows to graph elements. By the time data reaches the Continuous Query, it's a clean, typed change event.

Source Change Event Example

A PostgreSQL row insert mapped to a graph Node:


{
  "op": "Insert",
  "element": {
    "type": "Node",
    "labels": ["Order"],
    "reference": {
      "source_id": "pg-orders",
      "element_id": "Order:42"
    },
    "effective_from": 1709571234000,
    "properties": {
      "id": 42,
      "status": "delayed",
      "total": 149.99
    }
  }
}

Table name → Node label · Columns → properties · Primary key → element ID

Elements can be Nodes (mapped from table rows — table name becomes the label, columns become properties, primary key becomes the element ID) or Relations (mapped from foreign keys or explicit relationships). Insert and Update carry the full element (type, labels, reference, properties). Delete carries only metadata (reference and labels) — no properties needed since the element is gone. The query engine never touches raw database protocols.

Continuous Queries

Declarative queries that run perpetually in openCypher or GQL

Simple — find all delayed orders


MATCH (o:Order)
WHERE o.status = 'delayed'
RETURN o

Cross-source — orders with customer details from different databases


MATCH (o:Order)-[:PLACED_BY]->(c:Customer)
WHERE o.total > 1000
RETURN o, c.name, c.email
This is the core innovation. Queries run perpetually, maintaining a live result set. When source data changes, Drasi determines which results are added, updated, or deleted. Queries use openCypher or GQL — graph-native languages natural for relationship-rich data. Critical: queries can span multiple sources in a single statement. Order from Postgres, Customer from Cosmos DB, joined in one query. No shared schema or event bus required. If asked about streaming SQL (Materialize, ksqlDB): Drasi is for change detection across existing systems. Streaming SQL processes event streams. Different problem shapes.

Reactions

Act when query results change

Webhooks SignalR SSE Event Grid Debezium Dapr Stored Procs Gremlin Dataverse

One query → many reactions · One reaction → many queries

Reactions receive typed change events: added (new result matches), updated (result changed but still matches, with before/after), deleted (no longer matches). Clean, unambiguous — no parsing event payloads to figure out what happened. Flexible wiring: one query can feed many reactions, one reaction can watch many queries.

Patterns & Advanced Capabilities

Now let's look at some advanced patterns that differentiate Drasi from simpler change detection approaches.

Multi-Source Queries

  • Join data across heterogeneous systems in a single query
  • No shared schema or common event bus required

MATCH (o:Order)-[:CONTAINS]->(p:Product)
// Order from PostgreSQL, Product from Cosmos DB
WHERE p.inventory < 10
RETURN o.id, p.name, p.inventory
This is a key differentiator. Single query joins Order from PostgreSQL with Product from Cosmos DB. No shared schema, no event bus, no manual correlation. Drasi handles the cross-source join and emits changes when either side updates.

Temporal Capabilities

trueFor

"Alert me if a sensor stays above threshold for 5 minutes"

trueLater

"Check back in 30 minutes — did this order ship?"

Temporal functions enable time-aware reactive logic declaratively. trueFor: condition must remain true for a duration before triggering. trueLater: schedule a future check — "did this resolve within 30 minutes?" Both enable detecting the absence of change, which is critical for SLAs and deadlines.

Detecting the Absence of Change

"A patient's vitals haven't updated in 10 minutes.
Is the sensor offline — or is it an emergency?" 


MATCH (p:Patient)-[:HAS_MONITOR]->(m:Monitor)
WHERE m.lastReading IS NOT NULL
  AND trueFor(m.lastReading = m.lastReading, 10, 'minutes')
RETURN p.name, p.room, m.lastReading

Most event-driven systems simply can't do this.

This is often the "aha" moment. Traditional event-driven systems can't do this — there's no event when nothing happens. Drasi's trueFor detects when a value hasn't changed for a specified duration. Critical for healthcare, IoT, SLA monitoring. Let this moment land before the callout fragment.

AI-Driven Workloads

  • Keep embeddings & model inputs aligned with live data
  • React to the specific data changes that matter to your AI pipeline
  • No stale context — continuous freshness without polling
AI systems increasingly depend on fresh data. RAG pipelines need updated embeddings. Model inputs must reflect current state. Drasi detects exactly which data changed and triggers targeted updates — no polling entire datasets, no stale context.

Middleware Pipeline

Transform & enrich incoming changes before they reach your queries


// Reshape nested JSON with jq before it hits the query engine
{ "transform": ".payload | { id: .orderId, total: .items | map(.price) | add }" }

Also supports: Unwind · Decode · Relabel · Promote

Middleware transforms incoming changes before they reach queries. jq for complex transformations. Unwind to expand nested arrays into separate nodes. Decode for base64/hex strings. Relabel to normalize labels across heterogeneous sources. Pipeline is sequential — output of one feeds the next.

Event-Driven vs Change-Driven

Event-Driven

"Here's everything that happened.
Figure out what you care about."

Change-Driven

"Tell me exactly what you care about.
I'll tell you when it changes."
Backup slide summarizing the key distinction. Event-driven: producer defines events, consumers parse and filter. Change-driven: consumer declares what matters via query, platform delivers precisely those changes. Drasi implements the change-driven model.
